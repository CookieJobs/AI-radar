# Radar 浅调研框架（v0.1）— AI 产品经理方法论

> 这是 Radar Skill 的**核心方法论 IP**。
> 用户 agent 拿到这份框架 → 调自己的 LLM → 按方法论执行调研 → 输出报告。

---

## 第一部分：调研执行步骤（按顺序）

### Step 1: 信息收集

**目标**：基于调研边界，收集 5-10 条结构化事实。

**操作**：
- 用 web search（推荐：Exa / Tavily / Brave）/ 你能用的数据源
- 关键词：从 `core_question` 提取 2-3 个核心关键词
- 时间窗口：按 `time_window` 严格执行
- **至少 3 个不同信源**（避免单源偏差）
- 优先权威信源：官方博客 > 行业媒体 > 社交媒体

**输出**（先内部用，最后会重组）：
```
- 事实 1: [摘要] | 来源: [URL] | 时间: [date]
- 事实 2: ...
- 事实 3: ...
```

### Step 2: 事实摘要

**目标**：把 Step 1 收集的事实按 `focus_dimensions` 维度组织。

**操作**：
- 每个维度 200 字内
- 保留来源链接
- 不要做评价（评价留给 Step 3）

**输出结构**：
```markdown
## 关键事实

### [维度 1]
- 事实：[摘要]
- 来源：[URL]
- 时间：[date]

### [维度 2]
- ...
```

### Step 3: AI PM 5 维度评估（**Radar 核心方法论**）

**这是 Radar 区别于其他调研工具的关键**——不管调研什么对象，必须按这 5 个维度评估。

#### 维度 1：对用户（agent 主人）的具体影响

**问题**：基于用户跟对象的 `relationship`，这件事对用户的业务/工作有什么**直接**和**间接**影响？

**操作**：
- 短期影响（30 天内）：价格、能力、可获得性
- 长期影响（1 年）：竞争格局、技术路线、市场份额
- 不要泛泛而谈"很重要"/"值得关注"——给出**具体**影响

**输出模板**：
```markdown
### 1. 对你的具体影响
- **直接影响**：[一句话说清楚对用户业务/工作意味着什么]
- **短期影响**（30 天）：[具体会发生什么]
- **长期影响**（1 年）：[趋势性变化]
```

#### 维度 2：行业趋势信号

**问题**：这件事背后反映什么**更大趋势**？这个趋势的方向、强度、可信度如何？

**操作**：
- 不止说"是 AI 趋势"——给出**具体**的子趋势
- 评估强度（高/中/低）和可信度（高/中/低）
- 引用至少 1 个佐证

**输出模板**：
```markdown
### 2. 行业趋势
- **反映的子趋势**：[具体子趋势]
- **趋势强度**：[高/中/低]（基于市场动作数量）
- **趋势可信度**：[高/中/低]（基于参与者质量）
- **佐证**：[至少 1 个事实/数据]
```

#### 维度 3：你该采取的具体行动

**问题**：基于以上分析，用户**应该做什么**？

**操作**：
- 至少 3 条**可执行**建议
- 每条建议要明确：
  - 做什么（动词开头）
  - 为什么（关联上面的影响/趋势）
  - 时间窗口（什么时候做）

**输出模板**：
```markdown
### 3. 行动建议
- [ ] [行动 1]：[动词开头，10-20 字]
  - **为什么**：[关联分析]
  - **何时做**：[时间窗口]
- [ ] [行动 2]：...
- [ ] [行动 3]：...
```

#### 维度 4：未来 30 天怎么演化

**问题**：基于当前信号，**未来 30 天**这件事最可能怎么发展？什么信号点要观察？

**操作**：
- 给出**具体**的演化路径（不是"会发展"）
- 至少 2 个分支（理想/悲观）
- 给出**可观察的信号点**

**输出模板**：
```markdown
### 4. 30 天演化预测
- **理想路径**：[最可能的发展]
- **悲观路径**：[可能出现的风险/反向]
- **观察信号**：
  - [ ] [信号 1]：[具体可观察的事件/数据]
  - [ ] [信号 2]：...
```

#### 维度 5：风险点

**问题**：这件事有哪些**潜在风险**？风险的发生概率和影响如何？

**操作**：
- 至少 2 个风险
- 每个风险评估：**概率**（高/中/低）× **影响**（高/中/低）
- 给出**对冲建议**（可选）

**输出模板**：
```markdown
### 5. 风险点
- **风险 1**：[描述]
  - 概率：[高/中/低]
  - 影响：[高/中/低]
  - 对冲：[怎么做可以降低风险]
- **风险 2**：...
```

### Step 4: 输出 Markdown 报告

## 第三部分：质量自检 Checklist

输出报告前，agent 必须检查：

- [ ] 摘要是否 3 句话、每句包含独立信息？
- [ ] 关键事实是否每条都带来源 URL？
- [ ] 5 个维度是否都覆盖了？
- [ ] 行动建议是否每条都有"为什么"和"何时做"？
- [ ] 30 天预测是否给出具体信号点（不是模糊的"会发展"）？
- [ ] 风险点是否每个都评估了概率和影响？
- [ ] 报告语言是否避免"很重要"/"值得关注"这类无价值表述？
- [ ] 报告是否避免"标题党"风格（不夸大、不卖关子）？

## 第四部分：特别说明

### 4.1 浅调研 vs 深调研

| 维度 | 浅调研（¥0.49） | 深调研（¥0.98） |
|---|---|---|
| 问卷问题数 | 5-7 | 10-15 |
| 5 维度评估 | 简版（每维度 1 段） | 详版（每维度 2-3 段 + 引用） |
| 行动建议 | 3 条 | 5-7 条 |
| 信号点 | 2 个 | 4-5 个 |
| 风险点 | 2 个 | 3-4 个 |
| 报告长度 | 800-1200 字 | 1500-2500 字 |
| 适用场景 | 快速判断、明确决策 | 深度分析、写文章、投融资材料 |

### 4.2 不要做的事

- ❌ **不要**虚构数据/URL——所有引用必须真实
- ❌ **不要**超过调研时间窗口——30 天问题不要写 5 年前的事
- ❌ **不要**忽略用户的 `relationship`——不同关系解读视角不同
- ❌ **不要**做投资建议（"买入/卖出"）——只给信息
- ❌ **不要**做法律建议——只给信息
- ❌ **不要**做"营销"——沉稳、客观、不夸张

### 4.3 Radar 的边界

- Radar **不**替你做决策
- Radar **不**保证报告准确（基于公开信息整合）
- Radar **不**替用户 agent 做 LLM 调用
- Radar **只**提供方法论 + 模板 + 框架
