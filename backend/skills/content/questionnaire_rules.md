# Radar 调研问卷设计规则（v0.1）

> 这是 Radar Skill 的核心方法论 IP 之一。
> 用户 agent 拿到这份规则 → 调自己的 LLM → 生成具体问卷。

---

## 一、问卷设计原则（AI 产品经理视角）

**核心思想**：调研质量 = 边界清晰度 × 维度完整度 × 意图理解度。
好问卷不是问得多，是**问得准**。

### 1.1 三段式结构

**第一段（必问，1-2 题）：收窄范围**
- 明确调研对象的**类别**和**时间窗口**
- 没有这一段，后续所有问题都漂浮

**第二段（必问，2-3 题）：明确深度**
- 重点关注哪些**维度**
- 报告的**用途**和**受众**——这决定语言风格和详略

**第三段（必问，2-3 题）：理解意图**
- 用户跟调研对象的**关系**（决定解读视角）
- 用户希望回答的**核心问题**（决定报告主线）
- 用户的**特殊约束**（可选，比如"我要做投资决策"/"我要写文章"）

### 1.2 提问原则

1. **单选优先**——除非必须多选，多选会逼用户做"组合决策"
2. **选项互斥**——不能"过去 30 天"和"最近 1 个月"同时存在
3. **不预设答案**——不要问"你是否觉得 Anthropic 比 OpenAI 强"（带引导）
4. **可操作**——每个答案都该让 agent 知道下一步做什么
5. **简短**——单个问题不超过 30 字，选项不超过 8 个字

### 1.3 避免的提问方式

- ❌ "你对这个话题有什么看法？"（开放无边）
- ❌ "请描述一下你的需求"（用户不知道怎么回答）
- ❌ "用一句话总结 X"（除非真的是元问题）
- ❌ "你想深入了解哪些方面？"（已经在问调研的调研，套娃）

## 二、必问的 5 个核心问题

无论调研什么对象，必须问这 5 个（按顺序）：

### 范围类（Q1-Q2）

**Q1: 调研对象是哪一类？**
- 公司（基础模型 / AI 应用 / AI 基础设施 / 其他）
- 赛道（垂直行业 AI / 通用工具 / 硬件 / 其他）
- 技术（论文 / 框架 / 模型 / API / 其他）
- 产品（开源工具 / SaaS / 移动应用 / 其他）
- 政策（国内 / 国外 / 行业自律 / 其他）

**Q2: 时间窗口？**
- 过去 7 天（极强时效，看爆发期）
- 过去 30 天（标准，能看趋势）
- 过去 90 天（看季度变化）
- 过去 1 年（看长期方向）
- 不限时间（看历史脉络）

### 深度类（Q3-Q4）

**Q3: 重点关注哪些维度？（多选）**
- 融资 / 商业（钱、人、合作伙伴）
- 产品 / 功能（新发布、定价、用户体验）
- 技术 / 能力（benchmark、架构、训练方法）
- 团队 / 高管（人员变动、组织调整）
- 竞品 / 市场（玩家变化、市占率）
- 趋势 / 信号（行业方向、潜在变化）

**Q4: 报告用途？**
- 内部决策（自己/团队/老板）
- 投融资材料（FA/VC/创业者）
- 写文章 / 内容创作
- 个人了解 / 学习
- 客户提案 / 销售

### 意图类（Q5）

**Q5: 你跟调研对象的关系？**
- 直接竞品（互相替代）
- 上下游（依赖 / 被依赖）
- 同赛道但不同细分
- 暂不相关（纯粹好奇）

**附加 Q5b（可选）**: 调研后你希望回答什么核心问题？（一句话）

## 三、按问题类型加的问题

### 调研公司

- 这家公司对你的业务/产品的**威胁度**（高/中/低/无关）
- 你希望重点了解**创始人/CEO 视角**还是**产品视角**
- 是否关心**未公开信息**（融资细节、员工评价等灰色信息）—— **Radar 默认只用公开信息**

### 调研赛道

- 你想了解**中国市场**还是**全球市场**
- 关心**早期阶段**（天使/A 轮）还是**成熟阶段**（B 轮以后）
- 想知道**头部玩家**还是**新兴挑战者**
- 是否关心**退出案例**（被收购/上市）

### 调研技术

- 你的**技术水平**（能读懂论文 vs 只会用 API）
- 关心**原理**还是**应用**
- 是否关心**复现成本**（自己跑 vs 直接用）
- 是否关心**生态**（配套工具、社区）

### 调研产品

- 你的**角色**（用户/竞品/投资人/合作伙伴）
- 关心的**用户群**（个人/中小企业/大企业）
- 是否关心**商业模式**（订阅/按调用/广告）
- 是否关心**护城河**（技术/数据/品牌/网络效应）

### 调研政策

- 你的**行业**（金融/医疗/教育/泛互联网）
- 关心的**地区**（中国/美国/欧盟/全球）
- 关心**已生效**还是**草案/预期**
- 是否需要**合规建议**（Radar 只给信息，不给法律意见）

## 四、问题数量控制

| 调研深度 | 问卷长度 | 价格 |
|---|---|---|
| 浅调研 | 5-7 题 | ¥0.49 |
| 深调研 | 10-15 题 | ¥0.98 |

**超过 15 题 = 用户填不动了**。如果信息还不够，深调研应该靠"问题质量"补，不是"问题数量"补。

## 五、输出格式规范

用户 agent 拿到本规则后生成的问卷，必须符合：

```markdown
# {对象类型} 调研问卷（{深度} · Radar 浅/深调研）

> 你最初的问题：{user_initial_question}

## 问题 1: {问题}
- [ ] 选项 A
- [ ] 选项 B
- [ ] 选项 C
- [ ] 其他：____

## 问题 2: ...

---

💡 填完上面所有问题，把答案一次性告诉你的 agent。
agent 会再次调用 Radar Skill 拿到 AI PM 调研框架 + 报告模板。
```

**格式要求**：
- 用 markdown checkbox `- [ ]` 而非 input/select
- 选项**简短**（不超过 8 字）
- 不要用 emoji 装饰问题
- 不要用表格（agent 解析困难）

## 六、特别注意事项

### 6.1 不要问"你为什么想知道"这类元问题

元问题对用户没价值（"我想了解 Anthropic"——为什么？"因为我感兴趣"——结束）。Radar **不问元问题**，问**直接信息**。

### 6.2 不要"建议用户做更多调研"

Radar 不是新闻聚合器。Radar 假设用户已经**知道调研是有价值的**，只需要**好方法论**。

### 6.3 不要问"你的预算是多少"

Radar 调研**不收费差异化**——所有用户拿同一份方法论。**用户预算不影响调研质量**，不影响 Radar 的输出。

### 6.4 默认假设：用户是 AI 产品经理

Radar 设计的问卷**默认用户是 AI 产品经理或类似角色**（懂 AI 行业、有商业判断力、关心产品和趋势）。如果用户角色不同，agent 应该用更适合他的框架（不是 Radar 的）。
