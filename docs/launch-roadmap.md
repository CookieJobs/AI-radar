### 信源接入任务（**v0.2 关键补完**）

49 条种子事件只能撑 MVP 跑通，**撑不起 90 天持续运营**。必须有真实抓取。

**目标**：7 月 1 日前，**每天自动从 40 个信源抓取 → 筛出 5-10 条候选 → 入库到 SQLite**

**最小实现路径**：
1. RSS 抓取：`feedparser` 库扫所有信源 RSS
2. 去重 + 初步分类（关键词匹配 domain/category）
3. 写入 SQLite（替代当前 in-memory）
4. 每天 8:00 自动跑（cron）

**暂不实现**：
- LLM 自动解读 / 推演（**必须人写**，这是 Radar 核心价值）
- 自动打分 impact_level（**必须人定**）

时间投入：15-20 小时（一次性）

## 每日 / 每周 / 每月节奏

- 月底：调整定价 / 限速 / 文档

## 风险与应对

### 风险 1：MCP 推广困难

**症状**：上线 1 个月 < 5 个接入 agent

**对策**：
- 不只推 MCP，同时把 Radar 包装成 REST API + Python SDK + 一行命令 demo
- 找 KOL（AI 圈 KOL）合作
- 给 OPC 创业者提供"白手套接入服务"（你帮他们配）

### 风险 2：信源抓取不稳定

**症状**：RSS 失效 / 被限流 / 抓取脚本崩

**对策**：
- 多源 + 备份信源
- 自建简单 RSSHub（针对被墙 / 无 RSS 的信源）
- 失败重试 + 告警

### 风险 3：agent 钱包上线推迟

**症状**：微信 / 支付宝 agent 钱包延期

**对策**：
- **冷启动期完全免费，不依赖付费时间表**
- 先跑 MCP 推广 + REST API 接入
- 等钱包上线再切付费

### 风险 4：内容质量不稳

**症状**：agent 主人反馈"判断不准"

**对策**：
- 强制每条事件过质量 checklist
- 错的判断立刻发"纠正"事件（透明）
- 每月一次"事件库 review"，删掉低质 / 补高质量

## 退出条件

如果满足以下任一条件，认真考虑是否继续：

- 90 天后 < 10 个活跃接入 agent（PMF 没找到）
- agent 主人反馈质量持续下滑
- 你自己疲惫（这不该变成 burnout 项目）
- 商业道德风险出现（绝对不能碰的线）

**退出的姿态**：
- 公开停更（透明）
- 把数据资产 / 代码开源给社区
- 转做更小众的方向（如付费 1v1 行业咨询）
